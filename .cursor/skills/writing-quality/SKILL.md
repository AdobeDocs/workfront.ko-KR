# 쓰기 품질

Workfront 설명서 스타일 안내서 및 DQTI 원칙을 사용하여 Workfront 설명서를 검토하고 개선합니다.

## 워크플로

콘텐츠를 검토하거나 편집할 때:

1. 대상 파일 읽기
2. 아래 규칙 적용 — 문제를 직접 해결하거나 플래그 지정
3. 작성자의 의도와 기술적 정확성 유지
4. 자세한 용어, 구두점 및 프로시저 규칙은 [reference.md](reference.md)을(를) 참조하십시오.

## 음성 및 관점

- **초** 사용(&quot;다음을 수행할 수 있습니다...&quot;) 리더에게 주소를 알릴 때
- 수축을 대화 음조로 사용(안 함, 안 함, 안 함, 그렇습니다.)
- 모범 사례에 &quot;권장&quot;을 사용합니다. &quot;권장&quot;이나 &quot;다음과 같음&quot;이 아닙니다.
- 한 사용자 그룹에 대해 쓸 때 세 번째 사용자에 있는 다른 역할을 참조합니다(&quot;볼 수 있음... 그러나 Workfront 관리자는 다음을 제한할 수 있습니다...&quot;)
- 젠더 대명사를 사용하지 마십시오. 문장을 재구성하거나 &quot;they&quot;를 사용하십시오.
- 선택 항목에 슬래시 없음 — &quot;or&quot; 사용(&quot;he&quot; 대신 &quot;he or&quot;)

## 명확성(DQTI)

- 문장당 하나의 아이디어
- 가능하면 문장을 25단어 미만으로 유지하십시오.
- 배경 컨텍스트가 아닌 작업 또는 결과를 사용하여 리드
- 의미를 전달하는 가장 간단한 단어 사용(&quot;활용&quot; 이 아닌 &quot;사용&quot;, &quot;시작&quot; 이 아닌 &quot;시작&quot;, &quot;정보&quot; 이 아닌 &quot;관련&quot;)
- 지정 방지(&quot;만들기&quot;가 아닌 &quot;만들기&quot;, &quot;구성&quot;이 아닌 &quot;구성&quot;)
- 구체적이고 구체적인 언어 사용 — 모호한 용어(&quot;여러&quot;, &quot;다양한&quot;, &quot;일부&quot;)를 사용하지 마십시오.
- 이중 부정 사용 방지
- 선택적 대명사(&quot;that&quot;, &quot;who&quot;)를 사용하여 문장 구조에 명확성을 추가합니다.

## 구체성(DQTI)

- 추상적인 설명 대신 구체적인 예제 사용
- 예제에 실제 값 포함
- 정확한 UI 요소, 필드 및 영역의 이름을 지정합니다.

## 작업 방향(DQTI)

- 시스템 *is*&#x200B;이(가) 아닌 사용자가 *할 수 있는*&#x200B;에 집중합니다.
- 필수 동사를 사용하는 리드 절차(&quot;작업 만들기&quot;, &quot;알림 구성&quot;)
- 사용자가 행동하기에 충분한 컨텍스트를 제공하지만 더 이상 제공되지 않습니다
- 자세한 도움말에 대한 링크를 포함합니다(&quot;자세한 내용은 [article]&quot;).

## 대문자 사용

- **머리글**: 문장 대소문자가 항상 (&quot;작업 만들기&quot;가 아닌 &quot;작업 만들기&quot;)
- **Workfront 용어**: UI 요소를 직접 참조하는 경우에만 대문자화
   - 직접 참조: &quot;계획된 완료 일자 필드를 완료합니다.&quot;
   - 간접 참조: &quot;각 작업에는 계획된 완료 일자가 있어야 합니다.&quot;
   - 팁: 용어 앞에 &quot;the&quot; 또는 &quot;a&quot;를 넣을 수 있는 경우 일반적으로 소문자여야 합니다
- **단추**: UI 대소문자를 팔로우합니다. 단, 모든 대문자 단추는 → 대/소문자를 제외합니다.
- **역할**: UI에서 역할에 대한 &quot;시스템 관리자&quot;, 사용자에 대한 &quot;Workfront 관리자&quot;

## 제목

- 문장의 경우 명령형 명령 사용(&quot;작업 만들기&quot;, &quot;액세스 수준 구성&quot;)
- 머리글은 작업을 기반으로 해야 함 — 사용자가 수행할 작업을 설명합니다.
- 개요 문서는 &quot;개요&quot;로 끝남(&quot;프로젝트 개요&quot;)
- 명확한 하위 목표를 사용하여 긴 섹션을 여러 머리글로 나누기

## 상호 참조 및 링크

다음과 같은 정확한 패턴을 사용합니다.

| 상황 | 형식 |
|-----------|--------|
| 정보를 제공한 후 링크 | &quot;자세한 내용은 [문서 이름]을 참조하세요.&quot; |
| 이전 정보가 없는 링크 | &quot;자세한 내용은 [문서 이름]을 참조하세요.&quot; |
| 다른 문서의 섹션에 연결 | &quot;자세한 내용은 [문서 이름]의 [섹션 이름]을 참조하세요.&quot; |
| 동일한 문서의 섹션에 대한 링크 | &quot;자세한 내용은 이 문서에서 [섹션 이름]을(를) 참조하십시오.&quot; |

- 단락: 텍스트와 인라인
- 절차 단계: 단계 후 새 라인에서
- 여러 링크: 글머리 기호 목록 사용
- 화면 위치에는 &quot;오른쪽 위 모서리&quot; / &quot;왼쪽 위 모서리&quot;를 사용합니다.

## 볼드체 및 기울임체

- **굵게** 클릭 가능한 동작 및 UI 요소는 프로시저 단계에만 적용됩니다.
- 절차 단계 외부에서 UI 요소를 굵게 표시하지 않음
- 단계 외부에서 대화 상자 이름, 페이지 이름, 아이콘 이름 또는 메뉴 이름을 굵게 표시하지 않음
- 사용자가 입력해야 하는 텍스트에 *기울임꼴*&#x200B;을 사용합니다(&quot;URL 상자에 *my.workfront.com* 입력&quot;).
- 강조할 때 이탤릭체를 사용하지 말고 대신 문장을 재구성하십시오

## 참고, 팁 및 경고

제한적으로 사용 - 초과를 사용하면 독자가 볼 수 없습니다.

- 섹션당 최대 한 개의 메모/팁/경고
- 여러 메모를 스택하지 않음
- 글머리 기호 목록이 있는 하나의 메모로 관련 메모 결합
- 새 기능을 발표하는 데 노트를 사용하지 마십시오. 문서 텍스트를 대신 재작업하십시오.

| 유형 | 용도 |
|------|---------|
| `>[!NOTE]` | 단락에 맞지 않는 정보, 좌절 방지, 버전 관리/호환성 |
| `>[!TIP]` | 사용자의 생활을 편리하게 해 주는 제안 사항이 있으면 좋습니다. |
| `>[!IMPORTANT]` | 단계 또는 절차에 대한 중요한 정보 |
| `>[!WARNING]` | 사용자에게 잠재적인 데이터 손실에 대해 경고 |

## 절차

- 필수 명령 제목 사용(&quot;작업 만들기&quot;)
- 필요한 경우에만 소개 — 머리글로 충분합니다.
- 기본 도입 형식: 무한 구문 + 콜론(&quot;임시 암호를 만들려면:&quot;)
- 단일 단계 절차에서는 여전히 번호 매기기 목록을 사용합니다.
- &quot;type&quot; 또는 &quot;select&quot; 사용 — &quot;set&quot; 또는 &quot;specify&quot;와 같은 모호한 단어를 사용하지 않음
- 조건부 단계: &quot;(조건부) 두 팀 이상의 멤버인 경우 다음을 선택합니다...&quot;
- 옵션 단계: &quot;(선택 사항) 이모지를 추가하려면 클릭...&quot;
- 시스템 응답이 발생하기 전이나 그 직후에 설명
- 여러 방법: 가장 간단한 방법을 먼저 문서화한 다음 &quot;또는&quot;을 사용합니다.

### 절차의 직각 대괄호

- `>`을(를) 사용하여 메뉴/탭 탐색 표시: &quot;**전자 메일** > **알림**&quot;
- 대괄호가 아닌 단추 이름을 굵게 표시
- 대괄호 주위에 공백 포함
- 메인 메뉴 탐색에 대괄호를 사용하지 않음 — 메인 메뉴 코드 조각을 사용합니다.

## 구두점 빠른 참조

| 규칙 | 예 |
|------|---------|
| 옥스퍼드 쉼표 항상 | &quot;동기, 수단, 기회&quot; |
| 프로시저에서 &quot;then&quot; 앞에 쉼표 | &quot;설정 을 클릭한 다음 인터페이스 를 클릭합니다.&quot; |
| 문장 중간값의 쉼표(매년) | &quot;2016년 1월 2일, ...호.&quot; |
| 쉼표(월-년 전용) 없음 | &quot;2016년 1월&quot; |
| 목록 앞에 콜론, 뒤에 소문자 | &quot;다음 중 하나를 선택하십시오. 옵션 A&quot; |
| 시술 제목 후 결장 없음 | &quot;작업 만들기&quot;(&quot;작업 만들기:&quot; 아님) |
| Em 대시는 거의 사용하지 않음 | 가능한 경우 문장 재구성 |
| 파일 확장명의 마침표 | &quot;.pdf 파일 업로드&quot; |

문장 부호 및 용어 규칙 전체를 보려면 [reference.md](reference.md)을(를) 참조하십시오.

## 완전성(DQTI)

- 사용자가 이해하고 실행하는 데 필요한 모든 정보 포함
- 사용자에게 필요하지 않은 정보를 포함하지 않음
- 항상 세부 설명서에 대한 &quot;자세한 정보&quot; 링크를 제공하십시오
- 문서 시스템 기본값: 목록에 &quot;(기본값)&quot; 또는 문장에 설명

## 조직(DQTI)

- 점진적 공개가 있는 논리적 구조(참조 → 개요 → 세부 정보)
- 시간 정렬 페이지에 대한 최신 콘텐츠(예: 릴리스 노트)
- 섹션당 한 개의 주제
- 3개 이상의 병렬 항목에 목록 사용
- 구조화된 비교 또는 필드 설명에 표 사용
- 텍스트로 된 단락 피하기 — 쉽게 읽을 수 있는 청크로 나누기

## 문서 유형

| 유형 | 용도 | 제목 규칙 |
|------|---------|-----------------|
| 개요 | 컨텍스트, 다이어그램, 작업 방식 — 절차 없음 | &quot;개요&quot;로 끝남 |
| 방법 | 단계가 명확한 특정 작업 | 명령동사 |
| 시작하기 | 설정 정보 + 새 사용자에 대한 링크 | &quot;시작...&quot; |
| 참조 | 테이블의 필드 설명 | 수사명사구 |
