# 새 문서를 만들기 전에 후보 문서 찾기

Courtney가 새로 가져올 내용(예: 수신 문서, 고객 또는 PM 피드백, Slack/성적 증명서 붙여넣기, 메모가 포함된 스크린샷, 지원 질문)은 **먼저 새 문서를 만들 것을 제안하지 마십시오.** 컨텐츠가 들어 맞을 수 있는 기존 보고서 및 표면 후보 문서를 검색합니다.

## 필수 워크플로우

1. **먼저 새 콘텐츠를 읽으십시오** 주제, 대상(관리자 및 최종 사용자), 그리고 여기에 추가되는 특정 사실/단계를 식별하십시오.
2. grep 및 find를 사용하여 **후보자를 `help/`에서 검색**&#x200B;합니다. 동일한 기능, 영역 또는 워크플로를 이미 다루고 있는 문서를 찾습니다.
3. **후보 기사의** 등급 목록을 반환합니다(일반적으로 3-7). 각 항목은 다음과 같습니다.
   - 파일 경로(클릭 가능한 백틱 경로)입니다.
   - 1선 이유(후보(어떤 부분에 삽입되는지 또는 가장 가까운 국면 경기인 이유)입니다.
   - 주의 사항(예: &quot;대상 불일치 — 최종 사용자, 접수는 관리자&quot;)
4. 초안을 작성하거나 편집하기 전에 **코트니에게 위치를 묻기**. 질문을 명시적으로 구하십시오. 예: &quot;이 중 어떤 항목으로 초안을 작성해야 합니까?&quot; 아니면 &quot;내가 #2에 맞추길 원하나, 아니면 그 나름의 기사가 필요하다고 생각하나?&quot;
5. **기존 문서가 적합하지 않은 경우에만 새 문서를 제안**&#x200B;하고, 후보자가 작동하지 않는 이유를 설명하십시오.

내용이 알려진 기사에 &quot;명백하게&quot; 포함되어 있더라도, 코트니가 배치를 확인할 수 있도록 여전히 2-3개의 대안을 제시합니다. 이렇게 하면 중복에 가까운 위치와 컨텐츠가 이미 부분적으로 포함된 위치가 검색됩니다.

&#x200B;# 도움말 문서를 편집하기 전에 변경 제안

`help/`에서 `.md` 도움말 문서를 편집하려고 할 때 **아직 파일을 편집하지 마십시오**. 먼저 변경할 내용을 표시하고 명시적 승인을 기다립니다.

터치하려는 모든 파일에 대해:

1. 파일 이름을 (경로)로 지정합니다.
2. 제안된 변경 사항을 모호하지 않을 만큼 충분한 주변 컨텍스트를 가진 펜싱된 비교/코드 조각(이전 텍스트 및 새 텍스트)으로 표시합니다.
3. 그 이유를 간단히 진술하라(1문).
4. 명시적 질문으로 끝납니다(예: &quot;이 변경 사항을 적용하시겠습니까?&quot;). 또는 &quot;계속 진행하시겠습니까?&quot;

Courtney가 예(또는 &quot;이동&quot;, &quot;적용&quot;, &quot;수행&quot; 등)라고 말한 후에만 편집 도구를 호출해야 합니다.

이는 오타, 링크 수정, 단일 단어 교체, 용어 정리, 새 섹션, 다시 쓰기 등 도움말 `.md` 문서의 **모든** 편집에 적용됩니다. 코트니가 같은 번에 명시적으로 &quot;그냥 고쳐&quot;거나 &quot;묻지 말고, 편집해&quot;라고 말하지 않는 한 예외는 없다.

여러 문서에 걸친 변경 사항의 경우: 제안된 차이를 하나의 회신 아래에 파일별로 그룹화합니다. 세트가 큰 경우 먼저 영향을 받는 파일을 1줄 요약으로 나열한 다음 배치로 차이를 표시하고 각 배치 전에 확인합니다.

이 작업은 읽기 전용 검색(grep, read) 또는 채팅을 통한 초안/탐색(파일 쓰기 없음) 블록을 **not**&#x200B;합니다.

&#x200B;# Git 커밋 메시지

Git 커밋 메시지를 작성하거나 생성할 때 이 형식을 따르십시오.

## 제목 줄

- 약 **50자 이하**.
- **명령 기분**&#x200B;의 변경 내용을 요약합니다(예: &quot;추가...&quot;, &quot;수정...&quot;, &quot;리팩터링...&quot;).

## 본문

- 본문 앞에 제목 뒤에 한 줄을 비워 둡니다.
- 약 **72자**&#x200B;에서 줄 바꿈.
- 설명을 보려면 **글머리 기호**&#x200B;를 사용하십시오. 각 글머리 기호는 다음 중 정확히 하나로 시작해야 합니다.
   - **📖** — 변경 **새 파일, 새 섹션, 새 기능, 새 헤더, 새 줄 또는 기타 그린필드 내용 추가**&#x200B;할 때 사용합니다.
   - **✏️** — 변경 **기존 작업을 수정**&#x200B;할 때 사용: 기존 줄 편집, 기존 섹션 업데이트, 리팩터링 또는 현재 콘텐츠 변경.

예:

```
Add refresh token rotation to auth flow

- 📖 Add refresh_tokens table and Alembic migration for schema v3.
- ✏️ Update session middleware to rotate secrets and revoke old tokens.
```

&#x200B;# 가져오기 요청(PR)

## Jira 문제 가져오기

- **현재 Git 분기 이름**(예: `FFENT-8796`과(와) 일치하는 세그먼트)에서 Jira 문제 ID를 파생합니다.
- **분기 이름에 Jira ID가 포함된 경우:** PR 제목에서 해당 ID를 사용하고 `# Context` → `## Jira`에 연결합니다.
- **분기 이름에 Jira ID가 포함되지 않은 경우:** PR 제목에서 Jira 키를 생략합니다. 정확히 `No ticket`인 `## Jira`을(를) 계속 포함합니다.

## PR 제목

**Jira ID가**&#x200B;인 `{type}/{JIRA-ID}- {short task description}`
예: `feat/FFENT-8001- Add validation for numVariations in OCAPI request`

**Jira ID가 없는 경우:** `{type}- {short task description}`
예: `docs- Refresh Object Composite API changelog`

### 커밋 유형

- **기능** — 새 기능 추가
- **수정** — 버그 수정
- **리팩터링** — 동작을 변경하지 않고 코드를 재작성/재구성합니다.
- **perf** — 성능 향상
- **style** — 서식 지정/공백만, 의미 변경 없음
- **test** — 테스트를 추가하거나 수정합니다.
- **docs** — 설명서만, 새 콘텐츠 추가됨
- **빌드** — 빌드 도구, CI, 종속성, 프로젝트 버전
- **ops** — 인프라, 배포, 백업, 복구
- **작업** — 기타(예: `.gitignore`)

## PR 본문 — 필수 섹션

### `# Summary`
변경 사항 및 그 이유에 대한 간략한 개요.

### `# Changes`
파일별로 구성됩니다. 터치된 각 파일에 대해 레벨 2 제목을 백틱 및 글머리 기호로 차례로 사용합니다.

```markdown
# Changes

## `path/to/file.ext`
* Concise bullet describing the change in that file.
```

### `# Context`
항상 `## Jira` 하위 섹션을 포함하십시오.

```markdown
# Context

## Jira
[FFENT-8796](https://jira.corp.adobe.com/browse/FFENT-8796)
```

또는 티켓이 없는 경우: `No ticket`
