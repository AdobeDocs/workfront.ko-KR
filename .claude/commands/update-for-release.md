# 릴리스 업데이트(Workfront)

이 스킬은 예정된 기능 릴리스에 대한 Workfront 도움말 문서를 업데이트하는 과정을 안내합니다. 워크플로는 `remove-preview-highlighting`의 역버전입니다. 문서에 새 동작이 추가되고 미리 보기로 표시되며 다른 스킬이 정리합니다(나중에 GA 시).

## 범위

**모두**&#x200B;이(가) true인 경우 적용:

1. 사용자는 제공되는 기능에 대한 Workfront 도움말 문서를 업데이트 중입니다(일반적으로 미리보기 먼저).
2. 변경 사항은 GA 정리가 아닌 새로운 비헤이비어 또는 UI를 도입합니다. GA를 정리하려면 대신 **제거-미리 보기-강조 표시**&#x200B;를 사용하십시오.
3. 파일이 릴리스 노트가 **아님**&#x200B;입니다. 릴리스 노트의 경우 대신 **release-notes-formatter**&#x200B;을(를) 사용하십시오.
4. 사용자에게 기능 컨텍스트: 최소 짧은 설명과 스크린샷, 이상적으로는 PRD URL(Adobe Wiki)이 제공됩니다.

범위가 명확하지 않은 경우 시작하기 전에 확인하십시오.

## 필수 워크플로우(루프의 사람)

저장소를 **일괄 편집하지** 마십시오. 한 번에 한 개의 문서를 이동합니다. 각 기사 후에 다음 기사로 계속 진행할지 여부를 묻습니다.

### &#x200B;1. 기능 컨텍스트 수집

사용자와 확인:

- **변경 내용**(새 동작 또는 UI의 1-2문장 요약).
- 새 UI의 **스크린샷**. 제공된 경우 대상 문서의 `assets/` 폴더 아래에 설명 kebab-case 파일 이름으로 저장합니다(예: `add-custom-message.png`). 제공되지 않으면 대기 중인지 또는 자리 표시자 참조를 진행할지 여부를 묻습니다.
- 사용 가능한 경우 **PRD URL**(Adobe Wiki). `user-Adobe Wiki Confluence` MCP 도구 `get_wiki_content`을(를) 사용하여 가져오십시오. 이 설명서를 읽어 사용자가 UI에서 볼 수 없는 동작(알림 부작용, 나중에 항목을 편집하거나 추가할 때 발생하는 일, 표시되지 않은 문자 제한, 권한 등)을 찾습니다.
- **가용성**: 미리 보기 전용, 미리 보기 + 빠른 릴리스 또는 이미 GA입니다. 이렇게 하면 3단계에서 코드 조각을 선택할 수 있습니다.
- **명시적 제외**: 사용자가 건너뛸 모든 문서(예: &quot;이 기능은 템플릿에 없습니다&quot;).

### &#x200B;2. 영향을 받는 인벤토리 문서

기능 영역의 키워드(예: `approval workflow`, `document approval`, 특정 필드 레이블)를 사용하여 저장소를 검색합니다. 후보 목록 작성:

- 관련 `help/quicksilver/.../` 트리의 방법 문서.
- 기능 영역을 언급하는 개요 문서 및 FAQ.
- **제외** `product-announcements/`(릴리스 노트는 다른 기술을 사용).
- 다른 문서에 대한 링크 텍스트만 언급하는 경우 **TOC/인덱스 페이지를 제외**&#x200B;합니다.
- 사용자가 1단계에서 건너뛰도록 한 문서를 **제외**&#x200B;합니다.

사용자에게 후보 목록을 제공합니다. 업데이트할 항목과 건너뛸 항목을 묻습니다. 형제 문서가 누락된 것 같은 경우 상호 참조 `help/quicksilver/TOC.md`을(를) 참조하십시오.

### &#x200B;3. 미리 보기 코드 조각 선택

`help/_includes/snippets.md`을(를) 읽고 가용성별로 선택:

| 가용성 | 코드 조각 |
| --- | --- |
| 미리 보기 전용 — 강조 표시된 컨텐츠는 그렇지 않은 GA 문서의 새로운 기능입니다. | `{{highlighted-preview}}` |
| 미리 보기 전용 — 전체 문서가 새로워졌습니다. | `{{highlighted-preview-article-level}}` |
| 미리보기 + 빠른 릴리스 고객, 일반 | `{{preview-fast-release-general}}` |

현재 분기에 대해 릴리스 특정 코드 조각이 이미 있는 경우 일반 코드 조각보다 선호합니다. 적용하기 전에 사용자와 선택을 확인합니다.

### &#x200B;4. 문서에 따라 — 먼저 표시, 확인 후 편집

사용자 확인 목록의 각 문서에 대해:

1. **파일을 읽습니다.**

2. **강조 표시 패턴을 확인합니다.** 이 문서에 맞는 사용자에게 묻습니다(대답은 문서에 따라 다를 수 있음).

   - **섹션당 복제**: 기존 섹션 머리글에 `in Production`을(를) 추가합니다. `<div class="preview"> ... </div>`에 래핑된 `in Preview`이(가) 추가된 새 섹션을 추가합니다. 새로운 비헤이비어가 절차를 추가 단계, 새 이미지, 새 테이블 행 또는 다른 문구 등 의미 있게 변경하는 경우에 사용합니다. 방법 절차에 대해 일반적으로 사용됩니다.
   - **줄 바꿈**: `<span class="preview"> ... </span>`에 래핑된 기존 섹션 내부에 인라인으로 새 문장을 추가하십시오. 추가가 기존 단락, 표 셀 또는 FAQ 답변에 자연스럽게 맞는 한두 문장일 때 사용합니다.
   - **혼합**: 동일한 문서의 일부 섹션은 섹션 단위 중복을 사용하고 다른 섹션은 줄 단위 래핑을 사용합니다. 문서에 절차 섹션과 FAQ 스타일 섹션이 모두 있는 경우 이 옵션을 표시합니다.

3. **H1 제목 바로 뒤에 있는 줄에** 코드 조각을 배치합니다. 위아래에 빈 줄이 있습니다. 코드 조각이 소개 단락의 **이전**&#x200B;에 있습니다.

4. **새 세부 정보를 &quot;항상 포함&quot;과 &quot;검토 중&quot;으로 버킷하십시오.** 이것은 가장 중요한 단계입니다.

   - **항상 포함**(자동 적용, 프롬프트 없음): 사용자가 UI와 상호 작용하는 동안 관찰할 수 없는 보이지 않는 동작 예:
      - 부작용(예: &quot;이를 편집하면 모든 참가자에게 이메일이 다시 전송됨&quot;)
      - 다른 오브젝트 또는 이후 이벤트 간 비헤이비어
      - 사전 요구 사항 및 권한
      - UI에 표시되지 않는 제한
      - 사용자가 PRD, 문서 또는 제품 팀에서만 배울 수 있는 모든 항목
   - **검토 시작**(다중 선택 항목으로 `AskQuestion`을(를) 가진 사용자에게 제공): 기능을 사용하는 동안 사용자가 화면에서 볼 수 있는 팩트. 예:
      - UI에 이미 표시된 문자 카운터(예: `0 / 500`)
      - 필드의 기본 확장/축소 상태
      - 표시된 확인란의 기본 선택 상태
      - 필드 옆에 있는 단추 레이블
      - 인라인으로 표시되는 유효성 검사 메시지

   각 &quot;검토 중&quot; 항목에 대해 한 문장의 근거를 제공합니다(&quot;초보자에게 더 긴 메시지를 계획하도록 도움&quot;, &quot;이후 단계에서 이를 보지 못하는 사용자가 이를 확장할 수 있도록 도움&quot;). 사용자가 선택하는 항목만 포함합니다. 기본 원칙은 &quot;사용자가 작업을 수행할 때 화면에서 볼 수 있는 경우 다시 기술하지 않음&quot;이지만 사용자가 최종 호출을 받습니다.

5. **편집 제안.** 문서에 대한 발췌 전/후(또는 집중된 다른 스타일의 설명)를 표시합니다. 적용: 코드 조각 배치, 제목 변경, 새 미리 보기 콘텐츠 및 위치, 스크린샷 참조, 인라인 `class="preview"` 랩.

6. **명시적 승인을 기다리는 중**(&quot;okay&quot;, &quot;apply&quot;, &quot;yes&quot;).

7. **유효성 검사.** 작성한 후 파일에서 `ReadLints`을(를) 실행하고 문제를 보고합니다. 구조를 확인하려면 변경된 섹션을 다시 읽으십시오.

8. **쓰기 품질** 스킬에 대한 **산문 수준 편집 연기**. 음성, 대문자, 굵은 규칙 또는 링크 패턴을 여기서 다시 실행하지 마십시오. 산문 패스가 요청되면 `~/.cursor/skills/writing-quality/SKILL.md`을(를) 읽으십시오.

### &#x200B;5. 각 문서 이후

다음 문서로 이동할지, 현재 문서를 중지, 건너뛰지 또는 다시 방문할지 여부를 묻습니다.

### &#x200B;6. 세션 종료 — 릴리스 노트 복사/붙여넣기

사용자가 세션에 대해 래핑(&quot;완료&quot;, &quot;다름&quot;, &quot;중지&quot; 또는 다음 문서로 이어지지 않음)할 때 다음과 같이 질문합니다.

> &quot;개선 페이지에 대한 복사/붙여넣기 릴리스 노트 항목을 원하십니까?&quot;

그렇다면 1단계의 기능 컨텍스트와 이 세션에서 업데이트된 기본 도움말 문서를 사용하여 초안 항목을 생성합니다. **파일에 쓰지 않음** — 복사/붙여넣기 텍스트로만 제공하십시오.

**release-notes-formatter** 스킬에서 제품 영역 페이지 구조와 일치하도록 항목의 형식을 지정하십시오.

```markdown
## {Feature name}

>[!NOTE]
>
>Preview: {date or TBD}
>Production fast release: {date or TBD}
>Production for everyone: {date or TBD}

{1–3 sentences describing what changed and why it helps users. Lead with the benefit, not the UI action.}

For more information, see [{Primary article title}](/help/quicksilver/{path-to-article}.md).
```

규칙:

- 아직 알 수 없는 날짜에 `TBD`을(를) 사용하십시오. 사용자에게 날짜가 있는지 문의하십시오.
- 기능 이름은 문장입니다(첫 단어와 고유 명사만 대문자로 표시).
- 설명은 구현 세부 사항이 아니라 사용자가 이제 수행할 수 있는 작업에 중점을 두어야 합니다.
- 개요 페이지가 아닌 업데이트된 가장 구체적인 방법 문서에 대한 링크입니다.
- 모든 날짜를 알 수 없고 사용자가 자리 표시자를 원하지 않는 경우 `>[!NOTE]` 날짜 블록을 포함하지 마십시오. 생략하고 나중에 추가해야 합니다.

## 콘텐츠 규칙

### 제목

- 프로덕션 측 참조로 사용되는 기존 섹션 머리글에 정확히 **`in Production`**&#x200B;을(를) 추가합니다.
- 새 섹션 머리글에 정확히 **`in Preview`**&#x200B;을(를) 추가합니다.
- 제목의 나머지 부분을 대소문자 구분 안 함(`writing-quality`에 따라)으로 유지합니다.

### 래퍼 미리 보기

- **섹션 수준**: `<div class="preview"> ... </div>`에서 줄바꿈. 여는 태그와 닫는 태그를 각 태그의 위와 아래에 빈 줄이 있는 고유한 줄에 배치하면 Markdown 제목과 내부 목록이 계속 렌더링됩니다.
- **인라인(문장 수준)**: 기존 단락, 표 셀 또는 FAQ 답변 내에서 `<span class="preview"> ... </span>`에 줄바꿈을 적용합니다.
- `<div class="preview">` 내에 `<span class="preview">`을(를) 중첩하지 마십시오.

### 코드 조각 배치

- 코드 조각 줄은 H1 바로 뒤에 있으며 위와 아래에 빈 줄이 있습니다.
- 코드 조각이 소개 단락, `>[!IMPORTANT]` 설명선 및 모든 액세스 요구 사항 블록의 **이전**&#x200B;에 있습니다.
- 기사당 하나의 스니펫입니다.

### 스크린샷

- 문서의 `assets/` 폴더에 설명 kebab-case 파일 이름으로 새 스크린샷을 저장합니다.
- 새 미리보기 내 섹션 내에서 새 스크린샷을 참조합니다. 프로덕션 내 섹션의 스크린샷이 더 이상 기능을 정확하게 반영하지 않으면 그대로 두십시오. 이 스크린샷은 GA까지 프로덕션 동작을 나타냅니다.
- 스크린샷 파일 이름을 제작하지 마십시오. 스크린샷이 아직 제공되지 않으면 사용자에게 문의하십시오.

### 참고 사항 및 팁

- 섹션당 최대 `>[!NOTE]`(또는 `>[!TIP]`, `>[!IMPORTANT]`, `>[!WARNING]`). 기존 섹션에 이미 메모가 있는 경우 관련 새 콘텐츠를 스택이 아닌 글머리 기호 목록과 동일한 메모에 결합합니다.

### 하지 말아야 할 일

- `product-announcements/`에서 문서를 편집하지 마십시오.
- 일괄 편집하지 마십시오. 매번 명시적 승인을 받아 한 번에 하나의 문서를 편집하십시오.
- 관찰 가능한 UI 팩트를 사용자에게 먼저 표시하지 않고 포함하지 마십시오.
- 사용자가 명시적으로 지시하지 않는 한 `<!-- ... -->` HTML 댓글 내의 내용을 수정하지 마십시오.
- `author:` 또는 관련 없는 Frontmatter 필드를 변경하지 마십시오.

## 편집 내용을 제시하기 전에 품질 검사

- 코드 조각은 H1 뒤 줄에 한 번 나타나며 위아래에 빈 줄이 표시됩니다.
- 기존 섹션 머리글은 `in Production`(으)로 끝납니다.
- 새 섹션 머리글은 `in Preview`(으)로 끝나며 섹션은 `<div class="preview">` 내에 있습니다.
- 인라인 추가가 `<span class="preview">` 내에 있습니다.
- `ReadLints`이(가) 편집된 파일에서 깨끗합니다.
- 문서가 두 상태(미리보기 콘텐츠가 표시되고 숨겨짐)에서 올바르게 읽힙니다.
